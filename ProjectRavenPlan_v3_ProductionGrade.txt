Project Raven - Production-Grade Implementation Plan (v3)
================================================================

Core Principles:
1. LOCAL-FIRST: Free tier is a complete standalone desktop app. No backend dependency.
2. PRODUCTION-GRADE: No shortcuts, no deprecated APIs, no hacks. Built to last.
3. NATIVE PERFORMANCE: Use native modules for audio capture, not browser workarounds.
4. RELIABILITY: Must work flawlessly in real meetings. Zero tolerance for audio drops or lag.

================================================================================
PHASE A: Foundation (Local-First Architecture)
================================================================================
Goal: Electron app shell with proper architecture, persistence, and two-window system

A1 - Project Scaffold
  - Electron 28+ (latest stable)
  - React 18 + TypeScript 5 (strict mode)
  - Tailwind CSS with design system tokens
  - electron-vite for fast HMR
  - ESLint + Prettier (strict config)
  - electron-store for encrypted local persistence

A2 - Plan Mode System
  - Global config: mode: 'free' | 'pro' stored locally
  - Feature flags derived from mode
  - All conditional logic branches cleanly from this flag
  - Default: free (standalone, no backend)

A3 - Secure API Key Management
  - Onboarding flow collects Deepgram + Anthropic keys
  - Keys encrypted via electron-store (AES-256-GCM)
  - Keys validated on entry (actual API ping, not just format check)
  - Secure memory handling (keys never logged, cleared from memory when not needed)
  - Persisted across restarts, editable in settings

A4 - Two-Window Architecture
  - Main process as coordinator (no business logic in renderer)
  - Dashboard window: normal, resizable, light theme
  - Overlay window: frameless, always-on-top, transparent, stealth-capable
  - Robust IPC bridge with typed channels
  - Window state persistence (position, size)

A5 - Dashboard Shell
  - Header: "My Raven" branding, Mode dropdown, Stealth toggle, "Start Raven" button
  - Session list placeholder (grouped by date)
  - Settings gear icon
  - Light theme with proper contrast ratios (WCAG AA)

A6 - Overlay Shell
  - Pill toolbar: logo, stealth badge, mic toggle, device selector, minimize, close
  - Expandable response panel with tabs (Chat, Transcript)
  - Input bar with quick action chips
  - Dark/translucent theme with backdrop blur
  - Drag handle for repositioning

✓ CHECKPOINT A:
  - App launches reliably on macOS 12+ and Windows 10+
  - Two windows render with proper themes
  - API keys persist encrypted across restarts
  - Hotkeys work (Cmd/Ctrl+Shift+H toggle overlay, Cmd/Ctrl+Shift+R toggle recording)
  - No console errors or warnings (except expected Electron dev warnings)

PHASE B: Audio Engine (Production-Grade Capture)
================================================================================
Goal: Bulletproof audio capture with proper speaker separation

B1 - Native Audio Module Architecture
  - Create native Node addon using node-addon-api (N-API)
  - Separate implementations for macOS and Windows
  - TypeScript bindings for type safety
  - Graceful fallback if native module fails to load

B2 - macOS Audio Capture (ScreenCaptureKit)
  - Native Swift/Objective-C module using ScreenCaptureKit (macOS 12.3+)
  - SCStreamConfiguration for audio-only capture
  - Capture system audio WITHOUT showing picker dialog
  - One-time Screen Recording permission (guided onboarding)
  - 16kHz mono PCM output for Deepgram compatibility
  - Proper cleanup on stop (no orphaned streams)

B3 - Windows Audio Capture (WASAPI)
  - Native C++ module using WASAPI loopback capture
  - Capture system audio from default output device
  - No special permissions required on Windows
  - 16kHz mono PCM output
  - Handle device changes gracefully

B4 - Microphone Capture (AudioWorkletNode)
  - Modern AudioWorkletNode (NOT deprecated ScriptProcessorNode)
  - Worklet processor runs in separate thread (no main thread blocking)
  - Echo cancellation, noise suppression enabled
  - Device enumeration and selection
  - Handle permission denial gracefully with user guidance

B5 - Dual-Stream Architecture
  - Microphone stream → "You" (outgoing messages)
  - System audio stream → "Them" (incoming messages)
  - Separate Deepgram WebSocket connections per stream
  - Streams tagged at source, never mixed
  - Graceful degradation: if system audio fails, mic-only mode with diarization fallback

B6 - Audio Pipeline Manager (Main Process)
  - Coordinates mic + system audio streams
  - Routes audio chunks to appropriate Deepgram connection
  - Broadcasts state to both windows
  - Handles start/stop/error lifecycle
  - Chunk buffering for network jitter

✓ CHECKPOINT B1 (Audio):
  - Click record → mic captures YOUR voice (no picker)
  - System audio captures MEETING audio (no picker after initial permission)
  - Audio flows continuously (verify with chunk count logs)
  - Clean stop with no orphaned processes
  - Works on both macOS and Windows

PHASE B (continued): Transcription & AI Engine
================================================================================

B7 - Deepgram Integration (Dual Connection)
  - Two WebSocket connections: one for mic, one for system audio
  - Model: nova-3 (latest, best accuracy)
  - Language: 'multi' for auto-detection (supports code-switching)
  - Features: smart_format, punctuate, interim_results
  - Keep-alive pings every 8 seconds
  - Automatic reconnection on disconnect (exponential backoff)
  - Proper CloseStream on stop

B8 - Transcript Data Model
  - Each utterance stored as entry: { id, source: 'mic'|'system', speaker: 'you'|'them', text, timestamp, isFinal }
  - Entries stored chronologically
  - Interim results shown in italics, replaced when final
  - Full transcript text derived from entries ("You: ... / Them: ...")

B9 - Claude AI Integration (Intelligent Context)
  - Conversation history maintained: array of { role: 'user'|'assistant', content, timestamp }
  - Claude receives: system prompt + conversation history + current transcript + user request
  - Claude tracks what it already answered (no repetition)
  - Smart focus on NEW content since last response
  - Streaming responses (SSE) for real-time display
  - Rate limit handling (429 → backoff, retry)
  - Error handling with user-friendly messages

B10 - Quick Action System
  - Action chips: Assist, "What should I say?", Follow-up, Recap
  - Each chip = different prompt template (not system prompt — user message)
  - Custom text input for freeform questions
  - All actions include current transcript context
  - Response tagged with triggering action for UI display

B11 - Chat-Based UI (Overlay)
  - Two tabs: Chat | Transcript
  - Chat tab: scrollable message history (user requests + Raven responses)
  - Transcript tab: live transcript with You/Them bubbles (incoming/outgoing style)
  - Auto-scroll to latest message
  - Proper loading states (typing indicator while Claude thinks)
  - Error states (red toast, retry option)

B12 - Stealth Mode
  - macOS: setContentProtection(true) — hides from screen recording
  - Windows: SetWindowDisplayAffinity(WDA_EXCLUDEFROMCAPTURE)
  - Visual indicator in overlay (badge: "Visible" / "Hidden")
  - Toggle in both dashboard header and overlay toolbar
  - State persisted in settings

B13 - Global Hotkeys
  - Cmd/Ctrl+Shift+H: Toggle overlay visibility
  - Cmd/Ctrl+Shift+R: Toggle recording
  - Cmd/Ctrl+Enter: Trigger AI suggestion (when overlay focused)
  - Hotkeys work even when app not focused (globalShortcut)
  - Conflict detection (warn if hotkey already registered)

✓ CHECKPOINT B (Full Engine):
  - Start recording → mic AND system audio captured
  - Transcript shows You (right, cyan) vs Them (left, dark) correctly
  - Click "Assist" → Claude response streams in Chat tab
  - Click another action → Claude remembers previous response, focuses on new content
  - Stealth mode hides overlay from screen sharing (test with Zoom)
  - Hotkeys work globally
  - Stop recording → clean shutdown, transcript persists

PHASE C: Sessions (Persistent History)
================================================================================
Goal: Session history with robust local persistence

C1 - Session Data Model
  - Local SQLite database via better-sqlite3 (synchronous, reliable)
  - Schema: sessions (id, title, transcript_json, ai_responses_json, mode_id, duration_seconds, started_at, ended_at, created_at)
  - Indexes on started_at for fast sorting
  - Migration system for schema updates

C2 - Session Lifecycle
  - Start: Create session record with started_at
  - During: Append transcript entries and AI responses in memory
  - Stop: Write final data, calculate duration, set ended_at
  - Auto-save: Periodic flush to disk during long sessions (every 60s)
  - Crash recovery: Recover in-progress session on restart

C3 - Auto-Naming
  - After session ends, send transcript summary to Claude
  - Generate concise title (max 50 chars)
  - Update session record
  - Fallback: "Session at {time}" if Claude fails

C4 - Session History (Dashboard)
  - List grouped by date (Today, Yesterday, This Week, Earlier)
  - Each row: title, duration badge, timestamp, mode badge, three-dot menu
  - Menu: Copy transcript, Regenerate title, Export, Delete
  - Pagination for large history (virtual scrolling)
  - Search/filter by keyword

C5 - Session Detail View
  - Full transcript with You/Them styling
  - AI responses shown inline or in separate section
  - Session metadata (duration, mode, timestamps)
  - Edit title (click-to-edit)
  - Back navigation

C6 - Export
  - Export as Markdown (formatted with headers, timestamps)
  - Export as plain text
  - Export as JSON (structured data)
  - Copy to clipboard option

✓ CHECKPOINT C:
  - Sessions save automatically on stop
  - Dashboard shows history grouped by date
  - Click session → see full detail
  - Export works in all formats
  - Search finds sessions by keyword
  - Delete removes session permanently
