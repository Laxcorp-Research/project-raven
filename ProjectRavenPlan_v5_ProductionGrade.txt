Core Principles:
1. LOCAL-FIRST: Free tier is a complete standalone desktop app. No backend dependency.
2. PRODUCTION-GRADE: No shortcuts, no deprecated APIs, no hacks. Built to last.
3. NATIVE PERFORMANCE: Use native modules for audio capture, not browser workarounds.
4. RELIABILITY: Must work flawlessly in real meetings. Zero tolerance for audio drops or lag.
5. OS-NATIVE AEC: Use each platform's battle-tested voice processing (Apple/Microsoft).

KEY ARCHITECTURE DECISION: OS-Native Voice Processing (Not WebRTC)
--------------------------------------------------------------------------------

After extensive testing with WebRTC AEC3, we discovered that ScreenCaptureKit's
unpredictable latency (varies 50-200ms) makes manual AEC timing synchronization
extremely difficult. The solution: use each OS's built-in voice processing.

APPROACH COMPARISON:
┌─────────────────────┬──────────────┬─────────────┬─────────────────────────┐
│ Approach            │ Effectiveness│ Complexity  │ Notes                   │
├─────────────────────┼──────────────┼─────────────┼─────────────────────────┤
│ WebRTC AEC3         │ 90-95%       │ HIGH        │ Requires manual timing  │
│ (manual)            │              │             │ sync - fails with SCK   │
├─────────────────────┼──────────────┼─────────────┼─────────────────────────┤
│ Apple Voice Proc AU │ 95%+         │ LOW         │ OS handles everything,  │
│ (macOS)             │              │             │ hardware-optimized      │
├─────────────────────┼──────────────┼─────────────┼─────────────────────────┤
│ Windows Voice       │ 95%+         │ LOW         │ Same approach, Windows  │
│ Capture DSP         │              │             │ Media Foundation        │
└─────────────────────┴──────────────┴─────────────┴─────────────────────────┘

Decision: OS-Native Voice Processing
- Battle-tested at OS level (used by FaceTime, Teams, Zoom internally)
- Handles all timing automatically
- Hardware-optimized for each platform
- Brief calibration period (~3 seconds) then clean separation
- WebRTC libraries kept as fallback if needed

--------------------------------------------------------------------------------

B1 - Native Audio Module Architecture ✅
  - Swift executable for macOS (spawned as child process)
  - Rust module for Windows (to be implemented)
  - Binary stdout protocol for audio data streaming
  - TypeScript wrapper (nativeAudioBridge.ts)
  - Graceful fallback if native module fails to load

B2 - macOS System Audio Capture (ScreenCaptureKit) ✅
  - Native Swift module using ScreenCaptureKit (macOS 13.0+)
  - SCStreamConfiguration for audio-only capture
  - Capture system audio WITHOUT showing picker dialog
  - One-time Screen Recording permission (guided onboarding)
  - 48kHz stereo capture → downsample to 16kHz mono Int16 PCM
  - Proper cleanup on stop (no orphaned streams)
  - Located: src/native/swift/AudioCapture/

B3 - Windows System Audio Capture (WASAPI) 🔲 PENDING
  - Native Rust module using WASAPI loopback capture
  - cpal crate for cross-platform audio abstraction
  - Capture system audio from default output device
  - No special permissions required on Windows
  - 48kHz capture → downsample to 16kHz mono Int16 PCM
  - Handle device changes gracefully
  - N-API bindings for Electron integration
  - Located: src/native/windows/ (to be created)

B4 - Microphone Capture with AEC

B4a - macOS: Apple Voice Processing Audio Unit ✅
  - AVAudioEngine with Voice Processing enabled
  - setVoiceProcessingEnabled(true) - single line enables AEC + NS + AGC
  - Voice Processing returns 9-channel format at 48kHz (spatial audio)
  - Extract channel 0 (mono microphone) from multi-channel buffer
  - Linear resampling: 48kHz → 16kHz
  - Format conversion: Float32 → Int16 PCM
  - ~20ms audio chunks for real-time streaming
  - Brief calibration period (~3 seconds) for adaptive filter

B4b - Windows: Voice Capture DSP (Media Foundation) 🔲 PENDING
  - Windows equivalent to Apple Voice Processing
  - Media Foundation transform for AEC + NS + AGC
  - WASAPI input device capture
  - Same output format: 16kHz mono Int16 PCM
  - Used by Microsoft Teams, Skype, Discord on Windows

B5 - Binary Protocol (Swift/Rust → Node.js) ✅
  - stdout binary protocol: [source_byte][length_hi][length_lo][pcm_data]
    • 0x00 = system audio (for "Them" transcript)
    • 0x01 = microphone audio (for "You" transcript)
  - stderr for debug logging (prefixed with [System] or [Mic])
  - Node.js parser in nativeAudioBridge.ts
  - Chunk buffering for network jitter

B6 - Dual-Stream Architecture ✅
  - Microphone stream (AEC-cleaned) → "You" (outgoing messages)
  - System audio stream → "Them" (incoming messages)
  - Separate Deepgram WebSocket connections per stream
  - Streams tagged at source, never mixed
  - Transcript merging with 5-second sliding window

B7 - Audio Pipeline Manager (Main Process) ✅
  - Spawns native audio capture helper (Swift binary)
  - Parses binary stdout protocol
  - Routes audio chunks to TranscriptionService
  - Broadcasts state to both windows via IPC
  - Handles start/stop/error lifecycle
  - Clean process termination on stop

  - System audio captures MEETING audio
  - Audio flows continuously (verified with chunk count logs)
  - Clean stop with no orphaned processes

Key Changes from v4:
================================================================================

1. PIVOTED FROM WebRTC AEC3 TO OS-NATIVE VOICE PROCESSING
   - WebRTC AEC3 required manual timing sync
   - ScreenCaptureKit latency (50-200ms) made this unreliable
   - Apple Voice Processing handles everything automatically
   - Same approach for Windows (Voice Capture DSP)

2. SIMPLIFIED NATIVE MODULE ARCHITECTURE
   - Single main.swift file (not separate files per component)
   - Voice Processing returns 9-channel audio → extract channel 0
   - Linear resampling replaces complex audio converters

3. TRANSCRIPT MERGING
   - 5-second sliding window for temporal ordering
   - Both streams merged chronologically
   - Implemented in transcriptionService.ts

4. WINDOWS IMPLEMENTATION PLAN
   - Will use Voice Capture DSP (Media Foundation)
   - Rust standalone binary (same pattern as macOS Swift)
   - Development on actual Windows laptop
   - Estimated 2-3 days implementation

5. UPDATED PERFORMANCE BENCHMARKS
   - CPU usage < 10% (Voice Processing is efficient)
   - Brief calibration period (~3 seconds) is normal
   - No manual timing management needed

6. WEBRTC LIBRARIES KEPT AS BACKUP
   - Built and functional
   - Available if OS-native solutions have issues
   - May be useful for Linux support in future

================================================================================
LESSONS LEARNED
================================================================================

1. OS-native voice processing (Apple/Microsoft) is superior to manual AEC
   - Battle-tested, hardware-optimized
   - Handles timing automatically
   - Brief calibration period is acceptable

2. ScreenCaptureKit has unpredictable latency (50-200ms)
   - Makes manual AEC timing synchronization very difficult
   - Let the OS handle audio processing internals

3. Voice Processing returns multi-channel audio
   - macOS: 9 channels (spatial audio format)
   - Must explicitly extract channel 0 for mono mic

4. Development workflow matters
   - Test on actual hardware (not VMs for audio)
   - Cursor + Claude works great for native code
   - Same approach for Windows development

5. Binary protocol is robust
   - Simple format: [source][length][data]
   - Works reliably across Swift and Node.js
   - Will work same way for Rust
